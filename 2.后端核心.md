基于上面的 MailHub 项目架构，帮我实现后端核心模块，使用 Python + FastAPI。
数据层全部通过 src/store.py 读写 JSON 文件，不使用任何数据库和 ORM。

## 需要实现的文件（逐个完整输出）

### 1. backend/src/store.py
- 用 threading.Lock 保证并发安全
- 封装四个 JSON 文件的读写：
  - logs.json：append_log(entry) / get_logs(filters, page, page_size)
  - rules.json：get_rules() / add_rule(rule) / update_rule(id, rule) / delete_rule(id)
  - channels.json：get_channels() / update_channel(id, data)
  - stats.json：get_stats(date) / increment_stats(date, channel, success)
- 所有方法加 try/except，文件不存在时自动创建并初始化为空结构
- get_logs 支持按 status / from_addr / matched_rule / 日期范围筛选，支持分页

### 2. backend/src/receiver.py
- 使用 imap-tools 库连接 IMAP 邮箱
- 支持轮询模式（每 60 秒拉取一次未读邮件）
- 解析邮件字段：发件人、主题、纯文本正文（截取前 500 字）、附件名列表
- ParsedMessage 中额外保存 raw_message: email.message.Message 原始对象，
  供 EmailChannel 转发时直接使用，避免重新构建导致内容变化
- 拉取后标记为已读，记录日志

### 3. backend/src/router.py
- 从 config/routes.yaml 加载路由规则作为初始值
- 运行时规则以 rules.json（通过 store.get_rules()）为准，YAML 仅作人工备份
- 规则字段：name、match（from_contains / subject_contains / subject_regex，AND 关系）、
  targets（type + 参数）
- 返回匹配到的所有 target 列表
- 支持热重载（每次匹配前从 store 重新读取，无需监听文件）

### 4. backend/src/channels/base.py + email.py + wecom.py

base.py：
- BaseChannel 抽象类，有 send(message: ParsedMessage) -> bool 方法

email.py（EmailChannel）：
- 邮件转发必须原封不动，不得修改任何内容
- 使用 Python 标准库 email + smtplib 实现
- 直接转发原始邮件对象（email.message.Message），不重新构建邮件内容
- 保留原邮件的所有头部字段：Subject、From、Date、Message-ID、
  Content-Type、MIME 结构、所有附件
- To 字段替换为配置的收件人列表，其余头部一律不改动
- 不在邮件正文头部或尾部添加任何额外文字（如「此邮件由系统转发」之类）
- 不改变 Content-Type，HTML 邮件原样转发 HTML，纯文本原样转发纯文本
- 附件完整保留，不压缩、不重命名
- 转发时用 smtplib.SMTP_SSL 或 STARTTLS（根据端口自动判断，
  465 用 SMTP_SSL，587 用 STARTTLS，其他端口默认 STARTTLS）
- 支持多收件人（To 字段传入列表）
- 失败时记录完整错误日志，返回 False，不抛出异常

wecom.py（WeComChannel）：
- httpx 异步 POST 企微 Webhook
- 消息格式为 Markdown，内容包含：发件人、主题、正文摘要（前 200 字）
- 失败重试 3 次，退避 1/2/4 秒
- 失败时记录完整错误日志，返回 False，不抛出异常

### 5. backend/src/api/routes.py
- GET  /api/stats
  返回今日处理邮件数、成功数、失败数、各渠道分发量
  数据来源：store.get_stats(today)

- GET  /api/logs
  分页返回处理日志，支持按 status / 渠道 / 时间筛选
  数据来源：store.get_logs(filters, page, page_size)

- GET  /api/rules
  返回所有路由规则
  数据来源：store.get_rules()

- POST /api/rules
  新增规则，写入 rules.json
  调用：store.add_rule(rule)

- PUT  /api/rules/{id}
  更新规则，写入 rules.json
  调用：store.update_rule(id, rule)

- DELETE /api/rules/{id}
  删除规则，从 rules.json 移除
  调用：store.delete_rule(id)

- GET  /api/channels
  返回所有渠道配置

- PUT  /api/channels/{id}
  更新渠道配置

- POST /api/channels/{id}/test
  测试渠道连通性，调用对应 Channel.send() 发送测试消息

- POST /api/auth/login
  验证 ADMIN_EMAIL + ADMIN_PASSWORD（从 .env 读取）
  返回 JWT token，有效期 7 天

- WebSocket /ws/live
  实时推送每条邮件处理事件（JSON 格式）
  后端处理完一封邮件后广播到所有连接的客户端

## 数据模型（Pydantic，不用 SQLAlchemy）

ParsedMessage：
- id: str（uuid）
- received_at: datetime
- from_addr: str
- subject: str
- body_preview: str（正文前 500 字）
- attachments: list[str]（附件文件名列表）
- raw_message: email.message.Message（原始邮件对象，仅在内存中流转，不序列化入 JSON）

EmailLog（写入 logs.json 的结构）：
- 继承 ParsedMessage 除 raw_message 外的全部字段
- matched_rule: str | None
- targets: list[str]
- status: Literal["success", "fail"]
- error_msg: str | None
- duration_ms: int

## 统一响应格式
所有 REST 接口返回：
{"code": 0, "data": ..., "msg": "ok"}
错误时：{"code": 1, "data": null, "msg": "错误说明"}
用 FastAPI 的 JSONResponse 封装，不要用 HTTPException 直接抛出

## 其他要求
- 每个文件包含完整 import、类型注解、异常处理、logging 日志记录
- WebSocket 连接管理用一个全局 ConnectionManager 类维护连接列表
- JWT 用 python-jose 库，密钥从 .env 的 NEXTAUTH_SECRET 读取
- 所有 .env 变量通过 python-dotenv 在模块顶部加载，不要硬编码
- raw_message 字段在 Pydantic 模型中用
  model_config = ConfigDict(arbitrary_types_allowed=True) 允许非标准类型
- EmailLog 写入 JSON 前必须排除 raw_message 字段，
  用 model.model_dump(exclude={"raw_message"}) 序列化
